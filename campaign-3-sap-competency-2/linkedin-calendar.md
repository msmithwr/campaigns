## Post 10 — SAP Security on AWS: Zero Trust
**Week of 9 June | Category: Technical Deep-Dive**

**Title:** Zero Trust for SAP: Why Traditional Perimeter Security Is No Longer Enough

**Post Content:**

SAP environments contain some of the most sensitive data in any organisation — financial records, employee data, supplier information, customer details, and intellectual property. Yet many SAP landscapes still rely on perimeter-based security models that assume internal network traffic is trustworthy.

Zero Trust for SAP on AWS operates on a different principle: verify explicitly, use least-privilege access, and assume breach.

Here is what Zero Trust looks like for SAP on AWS:

🔐 𝗡𝗲𝘁𝘄𝗼𝗿𝗸 𝘀𝗲𝗴𝗺𝗲𝗻𝘁𝗮𝘁𝗶𝗼𝗻: SAP application servers, HANA databases, and web dispatchers in separate subnets with security groups enforcing least-privilege communication. No flat networks.

🔑 𝗜𝗔𝗠 𝗶𝗻𝘁𝗲𝗴𝗿𝗮𝘁𝗶𝗼𝗻: AWS IAM for infrastructure access control. Role-based access with MFA enforcement. No shared credentials, no standing administrative access.

🔒 𝗘𝗻𝗰𝗿𝘆𝗽𝘁𝗶𝗼𝗻 𝗲𝘃𝗲𝗿𝘆𝘄𝗵𝗲𝗿𝗲: Data encrypted at rest (EBS volumes, S3 backups) and in transit (TLS for all SAP communication paths). HANA data and log encryption enabled by default.

📋 𝗔𝘂𝗱𝗶𝘁 𝗮𝗻𝗱 𝗰𝗼𝗺𝗽𝗹𝗶𝗮𝗻𝗰𝗲: CloudTrail for API audit logging, GuardDuty for threat detection, Config for compliance monitoring. Every action recorded, every anomaly flagged.

🛡️ 𝗦𝗔𝗣-𝘀𝗽𝗲𝗰𝗶𝗳𝗶𝗰 𝗵𝗮𝗿𝗱𝗲𝗻𝗶𝗻𝗴: SAP user authorisation reviews, RFC security assessment, transport security, and gateway configuration — integrated with the AWS security layer.
